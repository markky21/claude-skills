# Vue Classical Testing

## Overview

Classical (blackbox) component testing for Vue 3. Test the rendering contract — props in, rendered text out. Everything else is an implementation detail.

**REQUIRED BACKGROUND:** classical-testing for general unit/integration testing principles (Khorikov's 4 quadrants, SuT/CuT pattern, mock boundaries). This skill extends those principles to Vue component rendering contracts.

**Sources (Vue.js official + Vue Test Utils docs):**
- "Test what a component does, not how it does it"
- "The more your tests resemble how your software is used, the more confidence they can give you"
- "The more you stub, the less production-like your test becomes"

## What to Test vs What Not to Test

**A component's public contract is: given these props, render this content.**

| DO test (rendering contract) | DON'T test (implementation details) |
|------------------------------|-------------------------------------|
| Rendered text given props | Button click -> emit wiring |
| Conditional rendering (brand present vs dash) | CSS classes / highlighting |
| Number of rendered items | Internal methods via `wrapper.vm` |
| Slot content | Sort/filter functions on the instance |
| Empty states | DOM structure or element ordering |

### Why NOT test click -> emit wiring?

Testing "click button X emits event Y with payload Z" is testing the **internal wiring** between a DOM element and a handler. It's an implementation detail because:
- It couples tests to DOM structure (button position, CSS selectors)
- The emit contract is already verified by TypeScript's `defineEmits`
- Refactoring the template (reordering buttons, changing wrapper elements) breaks tests without changing behavior
- This is integration/E2E territory, not component unit test territory

**Do NOT rationalize writing emit tests.** Common rationalizations:
- "Emits are part of the public API" — The **type interface** is public. The **wiring** is private.
- "Users click buttons" — Users don't inspect DOM order. Simulate real user flows in E2E tests.
- "We need to verify the component works" — Rendering tests verify it renders correctly. E2E tests verify it works end-to-end.

**If internal logic needs testing, extract it to a pure utility function and test it separately.**

```typescript
// Extract to utils/sorting.ts and test in isolation
export const sortByLabel = (a: Item, b: Item): number =>
  (a.label || '').localeCompare(b.label || '')
```

## Stubbing Strategy

**Use the simplest stub that works. Never replicate component behavior.**

| Situation | Stub |
|-----------|------|
| Component doesn't work in jsdom | `'el-button': true` |
| Need named slots (`#header`) | `{ template: '<div><slot name="header" /><slot /></div>' }` |
| Need scoped slot with row data | Custom `defineComponent` with provide/inject |
| Everything else | `true` |

**Boolean `true` stubs** (auto-generated by Vue Test Utils) handle props as attributes, `@click` as native fallthrough events, and default slots with `renderStubDefaultSlot: true`.

**Use `mount`, not `shallowMount`:**
- `shallowMount` over-stubs everything — breaks provide/inject chains, stubs dynamic components inside custom stubs
- Globally registered components (Element Plus) aren't auto-stubbed by `shallowMount` anyway — they're unresolved in the test environment
- `mount` with targeted stubs gives precise control

## Test Pattern — `cut` (Component Under Test)

Use a `cut` factory function that encapsulates **all** setup (mocks, mount, async flush) so each test is fully self-contained. No `beforeAll`, `beforeEach`, or shared mutable state.

```typescript
const defaultProps = {
  items: [createItem()],
  highlightIds: [] as string[]
}

const cut = async (propsOverrides: Record<string, unknown> = {}) => {
  // Inline any jsdom mocks the component needs
  globalThis.ResizeObserver = vi.fn().mockImplementation(() => ({
    observe: vi.fn(),
    unobserve: vi.fn(),
    disconnect: vi.fn()
  }))

  const wrapper = mount(Component, {
    props: { ...defaultProps, ...propsOverrides },
    global: {
      plugins: [ElementPlus],
      components: { Edit, Delete, Plus },  // icons used by the component
      stubs: { 'child-component': true }   // only stub children not under test
    }
  })
  await flushPromises()
  await nextTick()
  return wrapper
}

describe('ComponentName', () => {
  it('should render item labels from props', async () => {
    const wrapper = await cut({
      items: [createItem({ label: 'Apple' }), createItem({ label: 'Banana' })]
    })
    expect(wrapper.text()).toContain('Apple')
    expect(wrapper.text()).toContain('Banana')
  })

  it('should render dash when optional field is missing', async () => {
    const wrapper = await cut({
      items: [createItem({ brand: undefined })]
    })
    expect(wrapper.text()).toContain('\u2014') // mdash
  })
})
```

**Why `cut`?**
- Each test calls `cut()` directly — no shared mutable wrapper, no lifecycle hooks
- Mocks live inside `cut`, so tests are self-contained and order-independent
- Name `cut` (component under test) makes the factory's role immediately clear

## Red Flags — STOP and Delete

If you see ANY of these in your test, delete the test and reconsider:

| Pattern | Problem |
|---------|---------|
| `wrapper.vm.methodName()` | Accessing private methods (London school) |
| `wrapper.vm as unknown as { ... }` | Type-casting to reach internals |
| `wrapper.findAllComponents(StubComponent)` | Coupled to test infrastructure identity |
| `.trigger('click')` on action buttons | Testing click -> emit wiring (implementation detail) |
| `buttons[0]`, `buttons.find((_, i) => i === 1)` | Depends on DOM ordering |
| `expect(el.classes()).toContain(...)` | Visual behavior, not logic |
| Custom `defineComponent` + `h()` for el-button | Over-engineering — use `true` stub |
| `{ template: '<button @click="$emit(\'click\')">...' }` | Recreating component behavior in a stub |
| `beforeEach(() => { wrapper = ... })` | Shared mutable state between tests |
| `beforeAll(() => { ... })` | Hidden setup — put mocks inside `cut` factory |
| `describe('emit events')` or `describe('user interactions')` | Entire test sections dedicated to wiring |

**All of these mean: delete the test. Write a rendering contract test instead.**

## Element Plus in Tests

**Prefer using real Element Plus** via `global.plugins: [ElementPlus]` instead of stubbing individual components. This gives the most production-like test environment.

```typescript
import ElementPlus from 'element-plus'
import { Edit, Delete, Plus } from '@element-plus/icons-vue'

const cut = async (propsOverrides = {}) => {
  globalThis.ResizeObserver = vi.fn().mockImplementation(() => ({
    observe: vi.fn(), unobserve: vi.fn(), disconnect: vi.fn()
  }))

  const wrapper = mount(Component, {
    props: { ...defaultProps, ...propsOverrides },
    global: {
      plugins: [ElementPlus],
      components: { Edit, Delete, Plus },    // register icons used by component
      stubs: { 'child-component': true }     // only stub children not under test
    }
  })
  await flushPromises()
  await nextTick()
  return wrapper
}
```

**Key points:**
- `el-table` needs `ResizeObserver` mock in jsdom — put it inside `cut`, not in `beforeAll`
- Register icon components via `global.components` (they're separate from Element Plus plugin)
- Only stub **child components** that are not under test (e.g., expanded row, custom children)
- `await flushPromises()` + `await nextTick()` after mount — Element Plus renders asynchronously in jsdom

**Fallback stubs** — only when real Element Plus doesn't work for a specific component:
- Boolean `true` stubs work for nearly all Element Plus components
- Only `el-table` + `el-table-column` may need custom stubs (for scoped slot row data) if real Element Plus approach fails
- Only `el-card` may need a template stub (for named `#header` slot) if real Element Plus approach fails

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Stubbing Element Plus components individually | Use `global.plugins: [ElementPlus]` for real components |
| Custom `defineComponent` stubs for el-button, el-tooltip | Real Element Plus or `true` — never replicate behavior |
| `beforeAll`/`beforeEach` for mocks or wrapper | Put all setup inside the `cut` factory |
| `shallowMount` everywhere | `mount` with real Element Plus + targeted stubs for children |
| Testing sort/filter via `wrapper.vm` | Extract to pure utility, test separately |
| Testing click -> emit wiring | That's the component's private concern |
| Testing CSS classes for visual state | Leave to visual regression / E2E |
| Elaborate stubs replicating component logic | Stubs should be minimal, not faithful replicas |
